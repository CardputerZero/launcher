# 05 - 内置页面框架

本章说明 APPLaunch 当前内置页面的基类、启动边界、注册位置和生命周期约定。

## 1. 页面框架位置

共享页面框架位于 `ext_components/cp0_lvgl/include/ui_app_page.hpp` 和 `ext_components/cp0_lvgl/src/ui_app_page.cpp`。APPLaunch 在 `projects/APPLaunch/main/ui/launcher_ui_app_page.hpp` 中定义首页使用的 `home_base`，具体页面放在 `projects/APPLaunch/main/ui/page_app/`。

页面实现不再普遍是单个 header-only 类。复杂页面通常拆成：

- `ui_app_xxx.hpp/.cpp`：页面对象和生命周期。
- `ui_app_xxx_view.cpp`、`*_layout.cpp`：视图创建与布局。
- `model/xxx_model.hpp/.cpp`：可独立测试的状态和策略。
- `setting/`：设置页各子页面的模型、服务和视图。

构建时 `generate_page_app_includes.py` 将页面 include 汇总写入 `build/generated/include/generated/page_app.h`，不会在源码目录生成 `main/ui/generated/page_app.h`。

## 2. 基类与公共区域

`AppPageRoot` 管理页面 root screen、input group、返回回调和组件生命周期。`AppPage` 在此基础上提供常规应用页结构。`home_base` 是 APPLaunch 首页专用基类，为首页提供顶栏和内容区域。

页面通常通过以下接口参与框架：

| 接口 | 作用 |
| --- | --- |
| `screen()` | 返回页面 root screen |
| `input_group()` | 返回页面自己的 LVGL 输入组 |
| `navigate_home` | 请求返回 Launcher 首页 |
| `cp0_lvgl_start_app_page(page)` | 统一加载 screen 并切换 input group |

页面可按需要直接继承 `AppPageRoot` 实现全屏布局，但仍应遵守相同的启动和销毁边界。

## 3. 页面启动生命周期

内置页面由 `app` 的模板构造器启动：

```cpp
launch = [](Launch *owner) {
    if (!owner->begin_page_launch()) return;
    ui_loading::show("Loading...");
    lv_refr_now(nullptr);
    auto page = std::make_shared<PageT>();
    owner->app_Page = page;
    page->navigate_home = std::bind(&Launch::go_back_home, owner);
    ui_loading::hide();
    cp0_lvgl_start_app_page(*page);
};
```

关键约定：

- `LauncherPageLifecycleModel` 拒绝重复启动或重复返回。
- `app_Page` 在页面显示期间持有页面，返回首页后才释放。
- `cp0_lvgl_start_app_page()` 是加载 screen 和绑定 input group 的统一入口。
- 页面通过 `navigate_home` 返回，不应持有 `Launch` 的强引用。
- 异步 callback、timer、worker 和 signal 必须在页面析构前取消或失效化。

## 4. 当前注册的内置页面

当前首页注册事实以 `projects/APPLaunch/main/ui/builtin_app_registry.cpp` 的 `BUILTIN_APPS[]` 为准：

| 首页名 | 页面类或命令 | 条件 |
| --- | --- | --- |
| Python | `python3`，在 `UISTPage` 中运行 | 全平台，always-on |
| STORE | `appstore_exec` 外部程序 | 全平台，always-on |
| CLI | `UISTPage` | 全平台，always-on |
| GAME | `UIGamePage` | 全平台，always-on |
| SETTING | `UISetupPage` | 全平台，always-on |
| MATH | `calculator_exec` 外部程序 | 全平台，可配置 |
| LORA | `UILoraPage` | 全平台，可配置 |
| IP_PANEL | `UIIpPanelPage` | Linux 非 SDL，可配置 |
| SSH | `UISSHPage` | Linux 非 SDL，可配置 |
| TANK | `UITankBattlePage` | Linux 非 SDL，可配置 |

这里区分“编译进项目的页面代码”和“注册到当前首页的内置入口”。`page_app/` 中存在源码不代表它必然出现在 `BUILTIN_APPS[]` 或首页轮播中。

## 5. 注册新页面

1. 在 `main/ui/page_app/` 添加页面类，并让生成脚本能够发现所需 header。
2. 在 `builtin_app_registry.cpp` 的 `BUILTIN_APPS[]` 添加 `append_page_app<PageT>` 条目。
3. 为 `AppDescriptor` 提供 label、icon、config key、configurable 和 always-on 属性。
4. 若页面仅支持特定平台，用现有编译条件包围注册项。
5. 为 model、生命周期边界和失败路径补充测试，并在 SDL/设备目标验证页面进入和返回。

示意：

```cpp
{{"NEW", "new_100.png", "app_New", true, false},
 nullptr, false, true, false, append_page_app<UINewPage>},
```

应用设置菜单通过 `launcher_app_registry_entries()` 读取可配置 descriptor。切换后，注册表变更回调会触发 `Launch::applications_reload()`。

## 6. 嵌套视图与终端页

页面内部的子视图通常应复用当前页面 screen/input group，而不是将临时对象注册成首页应用。确实需要启动完整 `AppPageRoot` 时，也应使用共享页面切换帮助函数并明确 owner，避免手写两套 screen/group 状态。

`UISTPage` 同时用于 CLI 和 `Terminal=true` 动态应用。终端命令由 `Launch::launch_Exec_in_terminal()` 创建并持有 `UISTPage`，先调用 `cp0_lvgl_start_app_page()`，再调用 `exec()`。

## 7. 开发约定

- 业务状态和解析策略优先放入 `model/`，使其可在无 LVGL 环境下测试。
- LVGL callback 边界不得传播 C++ 异常。
- timer callback 必须验证 timer 和 owner 仍是当前实例。
- 资源路径通过 `launcher_platform::path()` 或共享资源服务解析。
- 页面销毁前停止 worker、watcher、signal 和异步操作。
- 不要把内置注册重新塞回 `launch.cpp`；该文件只负责编排和运行时状态。

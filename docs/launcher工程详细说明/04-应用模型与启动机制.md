# 04 - 应用模型与启动机制

本章说明 APPLaunch 如何统一内置页面、终端命令和外部程序。当前实现已拆分为应用编排、内置注册、`.desktop` 加载和目录监听四个模块。

## 1. 应用模型与模块边界

首页条目统一表示为 `app`：

```text
app
├── Name
├── Icon
├── Exec
└── launch(Launch *)
```

| 路径 | 职责 |
| --- | --- |
| `projects/APPLaunch/main/ui/launch.h` | `app`、`Launch` 和内置页面模板启动逻辑 |
| `projects/APPLaunch/main/ui/launch.cpp` | 当前选择、页面生命周期、三类启动方式、返回首页和重载编排 |
| `projects/APPLaunch/main/ui/builtin_app_registry.cpp` | 当前内置应用注册表、开关过滤和 exec 所有权判断 |
| `projects/APPLaunch/main/ui/desktop_app_loader.cpp` | `.desktop` 枚举、解析、安全校验和追加 |
| `projects/APPLaunch/main/ui/app_directory_watcher.cpp` | applications 目录 watcher 和 3 秒轮询 timer |
| `projects/APPLaunch/main/ui/model/launcher_navigation_model.*` | 内置页面启动/返回状态约束 |

`Launch::bind_ui()` 首次执行时注册应用开关回调，重建内置列表，追加动态应用并刷新轮播；随后启动目录 watcher 和长按 ESC 返回首页回调。重复调用只刷新轮播。

## 2. `Launch` 的当前状态

| 字段 | 说明 |
| --- | --- |
| `launch_page_` | 首页 `UILaunchPage` 的弱引用 |
| `current_app` | 当前中心应用索引，初始为 2 |
| `app_directory_watcher_` | applications 目录 watcher 及其 timer |
| `page_lifecycle_` | 防止页面重复启动、重复返回的生命周期模型 |
| `app_list` | 已启用内置项与合法动态项组成的列表 |
| `app_Page` | 当前内置页或终端页的生命周期持有者 |

轮播索引通过 `normalized_app_index()` 环形归一化。`carousel_slot_app()` 按中心索引计算五个槽位，不要求列表恰好有五项。

## 3. 当前内置注册表

注册表位于 `builtin_app_registry.cpp` 的 `BUILTIN_APPS[]`，不是 `launch.cpp`：

| 条目 | 类型 | 平台条件 |
| --- | --- | --- |
| Python | `python3` 终端命令 | 全平台 |
| STORE | `@appstore_exec` 外部程序 | 全平台 |
| CLI | `UISTPage` 内置页 | 全平台 |
| GAME | `UIGamePage` 内置页 | 全平台 |
| SETTING | `UISetupPage` 内置页 | 全平台 |
| MATH | `@calculator_exec` 外部程序 | 全平台，可配置 |
| LORA | `UILoraPage` 内置页 | 全平台，可配置 |
| IP_PANEL | `UIIpPanelPage` 内置页 | Linux 非 SDL，可配置 |
| SSH | `UISSHPage` 内置页 | Linux 非 SDL，可配置 |
| TANK | `UITankBattlePage` 内置页 | Linux 非 SDL，可配置 |

以 `@` 开头的 exec 通过 `launcher_platform::path()` 解析逻辑名称，例如 `appstore_exec` 和 `calculator_exec`。当前 STORE 不使用 `run_as_root`。

`launcher_append_enabled_builtin_apps()` 根据 `AppDescriptor` 和应用注册配置追加条目。前五个 always-on 条目形成默认首页顺序，中心初始为 CLI。

## 4. 内置页面启动与返回

内置页面模板执行以下流程：

```text
app.launch(owner)
  -> owner->begin_page_launch()
  -> 显示 Loading 并刷新
  -> make_shared<PageT>()
  -> 保存到 app_Page
  -> 设置 page->navigate_home
  -> 隐藏 Loading
  -> cp0_lvgl_start_app_page(*page)
```

`cp0_lvgl_start_app_page()` 是统一页面切换边界，负责加载页面 screen 和绑定页面 input group。业务代码不应重复手写 `lv_disp_load_scr()` 与 `lv_indev_set_group()` 组合。

返回首页时，页面调用 `navigate_home`。`Launch::go_back_home()` 先让 `LauncherPageLifecycleModel` 进入返回请求状态，再通过 `lv_async_call()` 调度 `lv_go_back_home()`；回调恢复 timer、显示首页、刷新并释放 `app_Page`。析构时会取消尚未执行的异步返回并停止 watcher。

## 5. 终端命令

`Terminal=true` 条目由 `Launch::launch_Exec_in_terminal()` 创建 `UISTPage`。它与普通内置页使用相同的生命周期门和 `cp0_lvgl_start_app_page()`，随后调用 `UISTPage::exec(exec)` 在 PTY 中运行命令。`Sysplause` 控制命令结束后是否等待用户确认。

CLI 自身也是 `UISTPage` 内置页；Python 则是指向 `python3` 的终端命令条目。

## 6. 外部独立程序

`Terminal=false` 条目由 `Launch::launch_Exec()` 启动：

```text
显示 Loading
  -> screensaver foreground = false
  -> LVGL_RUN_FLAGE = 0
  -> 解绑首页 input group
  -> 暂停 LVGL timer 并刷新
  -> cp0_signal_process_api({"ExecBlocking", exec, home_key_flag, keep_root})
  -> 恢复 timer、首页 input group 和首页 screen
  -> 隐藏 Loading并刷新
  -> LVGL_RUN_FLAGE = 1
  -> screensaver foreground = true
```

APPLaunch 当前通过进程 signal API 调用服务，不直接调用 `cp0_process_exec_blocking()`。底层设备/SDL 实现仍负责进程创建、降权、进程组和 ESC 退出策略。

## 7. `.desktop` 动态应用

`launcher_append_desktop_apps()` 使用文件系统 signal API 枚举 applications 目录，并只接受合法的 `.desktop` 文件名。每个文件最多读取 65536 字节，解析后还要通过以下检查：

- 必须包含合法的 `Name` 和 `Exec`。
- `DesktopExecIsSafe` 必须通过。
- `Exec` 不得与当前列表重复，也不得遮蔽内置应用 exec。
- `Icon` 必须能通过 `launcher_platform::path()` 安全解析。
- 最多追加 `LAUNCHER_MAX_DESKTOP_APPS` 个动态应用。

通过检查后，`Terminal` 和 `Sysplause` 决定使用终端模式还是外部程序模式。

`AppDirectoryWatcher` 通过 `WatchStart` 获取 watcher handle，创建 3 秒 LVGL timer，并用 `WatchPoll` 获取变化数量。发现变化时回调 `Launch::applications_reload()`，后者重新构建完整内置列表、重新扫描动态应用并刷新轮播。停止时先删除 timer，再调用 `WatchStop`。

## 8. 从输入到启动

```text
UILaunchPage 收到 Enter
  -> Launch::launch_app()
  -> app_at_index(current_app)
  -> selected->launch(this)
      -> 内置页：cp0_lvgl_start_app_page()
      -> 终端命令：UISTPage + exec()
      -> 外部程序：process signal ExecBlocking
```

修改应用系统时应保持这些边界：注册数据放在 `builtin_app_registry.cpp`，动态文件策略放在 `desktop_app_loader.cpp`，目录生命周期放在 `app_directory_watcher.cpp`，页面/进程切换编排留在 `Launch`。
